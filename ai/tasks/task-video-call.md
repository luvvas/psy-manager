# Task: Video Call ‚Äî Psychologist ‚Üî Patient (WebRTC + Magic Link)

Status: draft

## Goal

Enable a psychologist to start a video call from an appointment and share a
magic link that the patient opens in the browser ‚Äî no account required. Audio
and video run peer-to-peer via WebRTC. Signaling is handled by a WebSocket
endpoint added to the existing Hono API.

## Context

Relevant files:

- `apps/api/src/index.ts` ‚Äî entry point; WebSocket upgrade must be registered here
- `apps/api/src/routes/router.ts` ‚Äî tRPC root router; add `videoSessionRouter`
- `apps/api/src/db/schema.ts` ‚Äî add `videoSessions` table
- `apps/api/src/routes/` ‚Äî pattern for new tRPC route modules
- `apps/api/src/services/` ‚Äî pattern for new service files
- `apps/web/src/router.tsx` ‚Äî add public `/consulta/:token` route
- `apps/web/src/features/` ‚Äî new `consulta/` feature folder goes here
- `apps/web/src/components/layout/` ‚Äî psychologist layout; patient join page is layout-free

Existing behavior:

- Appointments exist in the `agendamento` module with a patient reference.
- There is no patient authentication and no patient-facing portal.
- Hono supports WebSocket upgrades via `@hono/node-server` (`upgradeWebSocket`).
- The API currently has no WebSocket endpoints.

## Requirements

### Database

- Add a `video_sessions` table to `apps/api/src/db/schema.ts`:
  - `id` (uuid, PK)
  - `appointment_id` (uuid, FK ‚Üí appointments, nullable ‚Äî sessions can be ad-hoc)
  - `psychologist_id` (uuid, FK ‚Üí psychologists)
  - `patient_token` (text, unique) ‚Äî random URL-safe token for the magic link
  - `status` (`pending` | `active` | `ended`)
  - `created_at`, `expires_at` (token expires after 24 h), `ended_at`
- Run a Drizzle migration.

### API ‚Äî tRPC procedures (`apps/api/src/routes/video-session.ts`)

- `videoSession.create` (protected): psychologist creates a session, receives
  `{ sessionId, patientJoinUrl }`. The join URL is
  `<VITE_PUBLIC_URL>/consulta/<patientToken>`.
- `videoSession.get` (protected): psychologist polls or fetches their active
  session by `sessionId`.
- `videoSession.end` (protected): marks session as `ended`, disconnects WS room.
- `videoSession.validateToken` (public): patient submits their token; returns
  `{ sessionId, psychologistName }` or a 404/410 error. No auth required.

### API ‚Äî WebSocket signaling server

Add a WebSocket handler at `GET /ws/signaling/:sessionId` in
`apps/api/src/index.ts` (or a dedicated `apps/api/src/ws/signaling.ts`
imported there).

Roles: each connection announces itself as `psychologist` or `patient` via the
first message after connecting.

Messages are JSON with a `type` field:

| type | direction | payload |
|---|---|---|
| `join` | client ‚Üí server | `{ role: "psychologist" \| "patient", token?: string }` |
| `ready` | server ‚Üí client | `{ peerId: string }` ‚Äî sent to each side when both are connected |
| `offer` | client ‚Üí server | `{ sdp: RTCSessionDescriptionInit }` |
| `answer` | client ‚Üí server | `{ sdp: RTCSessionDescriptionInit }` |
| `ice` | client ‚Üí server | `{ candidate: RTCIceCandidateInit }` |
| `offer` / `answer` / `ice` | server ‚Üí client | forwarded verbatim to the other peer |
| `peer_left` | server ‚Üí client | `{}` ‚Äî sent when the other side disconnects |

Server keeps an in-memory map `Map<sessionId, { psychologist?: WS, patient?: WS }>`.
When both sides are present, send `ready` to each and let them negotiate.
Validate the patient's `token` against the DB before admitting them.
Validate the psychologist's session ownership via a short-lived signed query
param (generated by `videoSession.create`).

### Infrastructure ‚Äî STUN / TURN

- STUN: use `stun:stun.l.google.com:19302` (free, no setup).
- TURN: document the setup of **coturn** on the existing EC2 instance in
  `step-by-step.md`. The API should read TURN credentials from env vars:
  - `TURN_URL` (e.g. `turn:turn.example.com:3478`)
  - `TURN_USERNAME`
  - `TURN_CREDENTIAL`
- The tRPC `videoSession.get` procedure should return the ICE server list
  `[{ urls: STUN }, { urls: TURN, username, credential }]` so the frontend
  never hardcodes credentials.

### Web ‚Äî Psychologist side (`apps/web/src/features/consulta/`)

- `useVideoSession` hook:
  - Creates a session via tRPC, stores `sessionId` and `patientJoinUrl`.
  - Opens a WebSocket to `/ws/signaling/:sessionId`.
  - Handles the full WebRTC offer/answer/ICE dance as the **offerer**.
  - Returns `{ localStream, remoteStream, status, patientJoinUrl, endCall }`.
- `ConsultaRoom.tsx` ‚Äî full-screen layout:
  - Large remote video (patient).
  - Small self-preview (picture-in-picture corner).
  - Copy-link button showing `patientJoinUrl`.
  - End call button.
- A "Iniciar videochamada" button on the appointment details view (or a
  floating action on the agendamento page) triggers session creation and
  navigates to `/consulta/:sessionId`.
- Route: `/consulta/:sessionId` ‚Äî protected, psychologist only.

### Web ‚Äî Patient side (public, no auth)

- Route: `/consulta/:token` ‚Äî public, skips `AuthGuard`.
- `JoinPage.tsx`:
  - Validates the token via `videoSession.validateToken`.
  - Shows psychologist name and a "Entrar na consulta" button.
  - On join: opens camera/mic permissions, connects WS, answers the WebRTC
    offer as the **answerer**.
  - Same full-screen `ConsultaRoom` layout (remote = psychologist).
- If token is invalid or expired, show a friendly pt-BR error page.

### Router changes (`apps/web/src/router.tsx`)

- Add `/consulta/:sessionId` inside the authenticated layout for the
  psychologist.
- Add `/consulta/:token` as a top-level public route (outside `AuthGuard`),
  using a bare layout (no sidebar).

## Constraints

- Keep user-facing text in pt-BR.
- Follow existing patterns in `apps/api/src/routes` and
  `apps/web/src/features`.
- Do not touch unrelated modules.
- Do not read or expose `.env` secrets; add new vars only to `.env.example`.
- The patient route must work without a Better Auth session.
- The in-memory WS room map is acceptable for a single EC2 instance. If
  horizontal scaling is needed later, replace it with Redis pub/sub ‚Äî note this
  as a known limitation in a code comment.
- Only one peer of each role per session. Reject a second connection of the
  same role.

## Acceptance Criteria

- Psychologist creates a session, copies the link, and sees their own camera.
- Patient opens the link in any modern browser without logging in, sees the
  psychologist's name, clicks join, and audio/video connects.
- Both sides see each other's video stream within ~3 seconds on the same
  network.
- Closing either tab sends `peer_left` to the other side and shows a
  disconnection message.
- Expired or invalid tokens show a clear pt-BR error.
- `bun run build` passes with no TypeScript errors.

## Known Limitations / Follow-up

- In-memory WS room map breaks on multi-instance deployments ‚Üí Redis pub/sub.
- No recording capability (out of scope).
- No canvas / annotation (separate task).
- coturn setup must be done manually on EC2 (document in `step-by-step.md`).
